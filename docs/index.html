01为什么需要这玩意？#

先讲故事，再讲技术。所有 SRE 系统都从一个具体的痛开始。

凌晨 3:17，你的手机被 PagerDuty 拍醒。屏幕上写着：

[ALERT]SEV-2
checkout-api: error_rate > 5% for 5m on prod-us-east
correlated: latency_p99 = 1840ms (baseline 220ms)
runbook: (empty)
oncall:  you

接下来你要做的事：

  1. 开 VPN，登 Grafana，找 checkout-api 的面板
  2. 看错误率从哪个版本起涨 → 翻部署历史
  3. 看日志里有没有 stack trace 或异常关键字
  4. 看 trace 里慢在哪个下游
  5. 翻团队 wiki 里有没有相关 runbook
  6. 判断要不要回滚、扩容、还是叫人

这套动作训练有素的 SRE 大约要 10-30 分钟。每多 1 分钟，丢的订单更多。每天值班 10 个告警，其中 7 个是误报，3 个是真问题——但你不知道哪个是哪个，每个都得查。

类比：医院夜班医生

急诊室一晚上来 100 个病人，大多数其实没事，但每个都不能拒诊。医生疲于奔命。一个好的"分诊系统"先 5 秒分级，把 80% 明显小毛病的扔给护士，把 20% 真急症送主治。
我们要做的，就是给 SRE 团队做这个分诊系统——但分的不是病人，是告警。

为啥不直接问 ChatGPT？

你可能想：把告警 + 日志一股脑塞给 GPT-4o，让它告诉我怎么办，不行吗？

试一下你就知道：

  • 它看不见你的数据：实时 metrics、traces、deploy 历史都在你公司内网
  • 它没有上下文：不知道 checkout-api 是不是核心服务，不知道你们的 SLO 是 99.9%
  • 它会乱编：没有团队 runbook 时，它会发明出听起来像那么回事但根本错的诊断
  • 它没有"工具"：不能主动去查日志、跑命令、回滚部署
  • 它不可观察：它说"我建议回滚"——为什么？基于什么证据？说不清

这就是为啥我们需要的不是"一个聊天机器人"，而是一个能调工具、有角色边界、产出可验证证据的多智能体系统。

03什么是 Agent？#

如果你完全没碰过 LLM agent，先把这一节啃完。

把"agent"翻译成中文别用"代理"，太抽象。我推荐叫"会主动调工具的 AI"。对比：

类型能干啥不能干啥例子
LLM 本身 读你的问题，按训练数据生成文字回复 看实时数据、调 API、写文件 ChatGPT 网页版
LLM + 工具调用 判断要不要调工具，告诉你"我想调 X 工具，参数是 Y" 真的执行那个工具——你的代码得真的去跑 OpenAI function calling
Agent =（LLM + 工具）× 循环，它自己决定下一步：调工具、看结果、再调下一个、出结论 跨越它的角色边界（如果你设定了的话） 我们的 log_detective
Multi-agent 多个 agent 协作，每个有不同角色和工具集 — 整个 SRE Agent 系统
伪代码：单 agent 长啥样 
while not_done:
    plan = llm.think(state)         # LLM 决定下一步
    if plan.is_final:
        return plan.answer
    result = call_tool(plan.tool, plan.args)
    state.append(result)            # 把工具结果塞回 LLM 上下文

那 "multi-agent" 加了什么？

多 agent 在单 agent 之上加了三件事：

  1. 角色（Persona）——每个 agent 有自己的 system prompt 和工具集。比如 log_detective 只会查日志，不会改部署
  2. 协作图（Graph）——谁先跑、谁等谁、消息怎么传。这是 LangGraph 干的活
  3. 共享状态（State）——所有 agent 写到同一个状态对象，但写法严格受控（schema + lane discipline）

04为啥 8 个不是 1 个？#

最常见的面试质问之一："一个大 prompt 不就行了，为啥搞这么复杂？" 来正面回答。

试试 "Single Big Prompt" 方案

设想我们走最朴素的路：

naive_v0.pyPYTHON
def diagnose(alert):
    prompt = f"""
你是 SRE。下面是告警 + 全部数据，给出根因和修复：
{alert.dump()}
{fetch_logs(alert.service)}
{fetch_metrics(alert.service)}
{fetch_traces(alert.service)}
{fetch_deploys(alert.service)}
{fetch_runbooks(alert.service)}
"""
    return llm.invoke(prompt)

这个方案有 5 个致命问题：

① 上下文爆炸

真实告警时拉一小时日志可能有 100MB。GPT-4o 上下文窗 128K token，你装不下。你只能预先做摘要——而"摘要什么"本身就是个决策，谁来做？

② 一个 prompt 难以专精

同一个 system prompt 既要懂日志正则、又要懂 PromQL、又要懂 Slack Block Kit、又要懂 git 回滚——LLM 会变成万金油，每个领域都半吊子。这是 LLM 工程师都吃过的亏。

③ 没法追责

它给了个根因，错了。是因为日志没看仔细，还是 metrics 解读错？单一 prompt 是黑盒。多 agent 时每个 agent 都有自己的 interpretation 字段——错了能定位到具体那一段。

④ 没法并行

查日志、查 metrics、查 traces 是互不依赖的，可以并发。单 prompt 顺序拼接，串行处理。多 agent + LangGraph 天然 fan-out，时间砍 5 倍。

⑤ 没法局部降级

Datadog 挂了，单 prompt 整个就报错。多 agent 系统里，metrics_analyst 返回 NO_SIGNAL，hypothesis_generator 拿剩下 4 路证据照样工作。

医院类比再现

急诊里你不会让一个医生既是放射科、又是心内、又是急诊外科、又是麻醉。各科室专精 + 分诊台协调，是几百年医学界打磨出来的组织结构。
多智能体系统其实就是把这个结构搬进软件——每个 agent 是一个"专科医生"。

那为啥不是 80 个？

另一个极端：把每个动作都拆成 agent，"日志关键词匹配 agent"、"trace span 排序 agent"、"deploy diff agent"……

这就掉进了 over-engineering 的坑。判断标准：一个 agent 是否对应一个清晰的"专业角色"？ 这个角色你能不能在 LinkedIn 上找到对应的真人岗位？

  • "Log Detective" → 真实岗位有：Log Analyst / SIEM Engineer ✓
  • "Metrics Analyst" → 真实岗位有：SRE / Observability Engineer ✓
  • "日志关键词匹配 agent" → 不是岗位，是函数 ✗

8 个 agent 对应的是 SRE 团队里能区分出来的 8 种岗位心智——再细就是函数，再粗就是万金油。

05架构总览：hub-and-spoke#

为啥是这个形状不是别的？4 种常见拓扑对比。

拓扑形状优点缺点用在哪
Chain
链式		 A → B → C → D 简单，调试容易 串行，慢；中间错一步全挂 简单 ETL
Mesh
全连接		 每个 agent 可调任何 agent 最灵活 失控、循环、调试不可能 论文 demo
Supervisor

SRE 事故诊断有个特点：流程是固定的——拿到告警 → 查 5 路证据 → 提假设 → 出修复。这跟"写一篇文章"那种发散任务不一样。所以我们选 hub-and-spoke。

① incident_pm——派活的

第一个跑。它读告警，做三件事：

  1. 提取关键元数据：service、severity、started_at
  2. 判断需要哪些证据通道（默认全开 5 个）
  3. 写一句一行的事故摘要给 Slack 用

关键设计：PM 不调用任何数据源——它只是路由器和摘要器。这样保证后续 5 个 worker 拿到的 prompt 是一致的，可以并行。

② log_detective——查日志的

典型流程：

log_detective 工作流FLOW
1. 从 provider 拉过去 15 分钟日志（Loki / Datadog / Mock）
2. 按 ERROR/WARN 关键字过滤
3. 聚合：top 错误信息 + 每分钟错误数 + 异常类型
4. LLM 把聚合结果转成一句话自然语言（"connection pool exhausted spike at 03:17, 240 errors/min"）
5. 返回 LogsEvidence(status="FOUND", peak_at=..., top_messages=..., interpretation=...)

注意第 4 步——LLM 只是把结构化数据翻译成人话，不让它做"判断"。判断是 hypothesis_generator 的活。这就是 lane discipline 落到代码上。

③④⑤ Metrics / Traces / Deploys——同款心智

三个 worker 结构相同，只是数据源和 schema 不同：

  • metrics_analyst: PromQL 拉 5 个核心指标，判断哪个 verdict=spike
  • trace_reader: 取 hot span，看 error_rate 是不是在某个 span 集中
  • deploy_historian: 看告警时间前 60 分钟有没有部署，看 commit 内容

⑥ runbook_consultant——团队大脑

这个特殊。它不调 LLM。

为啥不用 LLM？

它的工作是 "看到 alert，从团队 wiki 里找最相关的 runbook 章节"。这是纯检索任务（向量相似度），用 LLM 来做反而：(1) 慢 (2) 烧钱 (3) 可能幻觉出根本不存在的 runbook。
我们用 RAG——把 markdown runbook 切块、向量化（OpenAI / Ollama embedding 或退化到 TF-IDF）、按 alert 查询返回 top-K 段落。详见第 8 节。

⑦ hypothesis_generator——合证据

这是真正"思考"的 agent。它拿到 5 路证据，权重综合出最可能的根因 + 备选项。

关键 prompt 设计：

  • 所有 5 类证据按统一模板塞进 prompt
  • 明确要求"每个假设必须标注它基于哪几路证据"——这是 trust-but-verify 的基础
  • 用 with_structured_output(HypothesisList) 强制 Pydantic 输出，不接受自由文本
  • 如果 LLM 输出不合法 schema → 抛异常 → catch → 走 rule-based fallback

⑧ remediation_suggester——出修复

拿 hypothesis + runbook 引用，提取具体可执行命令：

  • "回滚 commit abc123：kubectl rollout undo deployment/checkout-api"
  • "重启 Redis 连接池：POST /admin/pools/redis/restart"
  • "扩容到 8 副本：kubectl scale --replicas=8 deployment/checkout-api"

步骤必须引用 runbook 路径作为依据。没引用的话 confidence 自动降级。

07类型化证据：让 LLM 输出"可验证"#

这是整个系统不胡说八道的最关键一层。

LLM 默认输出自由文本。问题大了——下游怎么 parse？怎么 assert？怎么单测？

答案：Pydantic schema 全程约束。每个 agent 的输出都是一个严格 schema 的 Python 对象。

src/sre_agent/schemas.py（节选）PYTHON
class EvidenceResult(str, Enum):
    FOUND      = "FOUND"       # 找到了真信号
    NO_SIGNAL  = "NO_SIGNAL"   # 这一路没东西
    ERROR      = "ERROR"       # 调失败了（数据源挂了等）

class LogsEvidence(BaseModel):
    source: Literal["logs"] = "logs"
    result: EvidenceResult
    hits: int = Field(0, ge=0)
    peak_at: datetime | None = None
    top_messages: list[str] = Field(default_factory=list, max_length=5)
    interpretation: str = Field("", max_length=400)

class Hypothesis(BaseModel):
    title: str = Field(..., max_length=120)
    detail: str = Field(..., max_length=800)
    confidence: float = Field(..., ge=0.0, le=1.0)
    supporting_evidence: list[Literal[
        "logs", "metrics", "traces", "deploys", "runbooks"
    ]] = Field(default_factory=list)
    contradicting_evidence: list[Literal[...]] = Field(default_factory=list)

这做对了 4 件事：

① LLM 输出强制结构化

LangChain 提供了 llm.with_structured_output(Hypothesis)——它在 prompt 末尾自动加 JSON schema 说明，并在收到响应后用 Pydantic 解析。不合法直接抛异常。

② Confidence 不能瞎填

confidence: float = Field(..., ge=0.0, le=1.0)——LLM 想给个 1.5 都不行。它必须给一个真正 0-1 的概率。

③ "证据引用" 是 enum，不是字符串

注意 supporting_evidence: list[Literal["logs", "metrics", ...]]——LLM 不能编造一个 "我用了 customer_feedback 通道" 这种不存在的证据来源。

④ 测试可以断言

tests/test_graph.py（典型断言）PYTHON
result = graph.invoke({"alert": alert}, config=...)
report = result["report"]

assert report.phase == "diagnosed"
assert report.logs.result == EvidenceResult.FOUND
assert report.logs.hits > 0
assert "redis" in report.hypothesis.top.title.lower()
assert "runbooks" in report.hypothesis.top.supporting_evidence

没有 schema 的话，上面这些 assertion 一个都写不出来——你只能 assert "redis" in some_string，脆得很。

面试金句

"Pydantic 不是给程序员看的，是给测试看的。一个没法被严格断言的 LLM 系统是没法上生产的。"

08护城河：Runbook RAG#

为啥这层是"团队大脑"——也是面试问到必拿高分的地方。

什么是 runbook

SRE 团队对"已知问题"的应对手册。比如：

runbooks/checkout-api.md（示例）MARKDOWN
## Redis 连接池耗尽（部署后）

> service: checkout-api
> tags: redis, deploy, connection-pool

### 症状
- error_rate 在部署后 2-5 分钟内突增到 5% 以上
- 日志里大量 `connection pool exhausted: timeout`
- p99 latency 飙到 1500ms+

### 根因
新版本默认 pool size = 16，老版本是 64。
HikariCP 配置改动遗漏。

### 修复
1. 立刻回滚到上一个 commit
2. 或者临时改 ConfigMap 把 pool size 调回 64
3. 长期：把 pool size 加进 SLO 监控

团队 wiki 里通常已经有几十甚至上百份这种文档。问题是没人在凌晨 3 点会去翻它——找不到、不知道存在、关键字搜不到。

RAG 怎么解决

RAG = Retrieval Augmented Generation。3 步：

RAG = 离线建索引 + 在线检索 + 把检索结果塞进 LLM prompt

三层 embedding 后端

我们没有把项目绑死在任何一个向量库或 embedding 提供方上：

层实现质量速度什么时候用
① OpenAItext-embedding-3-small★★★★★API 几十 ms有 OPENAI_API_KEY
② Ollama 本地nomic-embed-text★★★★本地几 ms有 Ollama 跑着
③ TF-IDF 关键词自实现，零依赖★★极快啥都没有，也能跑

三层逐级降级。这意味着哪怕你完全离线、没 OpenAI key、没 Ollama，runbook 检索还是能跑——只是质量从语义检索退化到关键词匹配。系统永远不挂。

src/sre_agent/nodes/runbook_consultant.py（核心）PYTHON
def runbook_consultant(state: GraphState) -> dict[str, Any]:
    alert = state["alert"]
    store = get_store()       # singleton, 启动时建好
    if store.size == 0:
        return _emit(RunbookEvidence(
            result=EvidenceResult.NO_SIGNAL,
            library_size=0,
        ))
    query = _build_query(alert)           # 关键词组合
    hits = store.search(query, service=alert.service, k=3)
    if not hits:
        return _emit(RunbookEvidence(result=EvidenceResult.NO_SIGNAL, ...))
    ev = RunbookEvidence(
        result=EvidenceResult.FOUND,
        hits=[RunbookHit(...) for h in hits],
        library_size=store.size,
        backend=store.backend_name,
    )
    return _emit(ev)

整个文件不到 80 行，不调任何 LLM。这就是 lane discipline 落到代码上——它的角色是"检索员"，不是"诊断师"。

09怎么保证不胡说#

LLM 系统能上生产的核心障碍。我们的 5 道防线。

从内到外 5 层防御——每一层都拦住一类失败模式

① Pydantic schema 强制结构化

已经讲过——LLM 输出不合 schema 直接报错。不接受自由文本作为 hypothesis。

② Lane discipline——角色不越权

每个 agent 的 system prompt（personas/*.md）里写明：

  • log_detective："只描述日志看到了什么，不做根因判断"
  • incident_pm："你不写代码、不给修复建议、不下结论"
  • hypothesis_generator："你不调任何数据源——只综合已有证据"

违反 lane 的输出会被发现：比如 log_detective 想塞个 "建议回滚" 进 interpretation 字段，pytest 里有 assertion 拒绝。

③ 证据引用是 enum

已经讲过——LLM 没法引用不存在的证据来源。

④ Confidence threshold

这是个绝妙的设计：

graph.py 终态判定PYTHON
phase = "diagnosed"
if hypotheses is None or not hypotheses.hypotheses:
    phase = "failed"
elif hypotheses.top.confidence < 0.4:
    phase = "no_signal"     # 诚实地说"我没把握"

当模型自己说"我对这个诊断只有 30% 把握"——系统不会假装 diagnosed，而是降级为 no_signal，把不确定性透传给人类。比假装高置信度 << 错诊更安全。

⑤ Rule-based fallback——LLM 全挂时的兜底

每个调 LLM 的 agent 都包了 try/except，失败时走规则：

hypothesis_gen.py fallbackPYTHON
def _fallback_from_evidence(state, error: str) -> dict:
    # 没有 LLM 时，用启发式拼一个诊断
    bullets = []
    if state.get("deploys") and state["deploys"].relevant_deploys:
        bullets.append(f"recent deploy ({state['deploys'].relevant_deploys[0].commit})")
    if state.get("runbooks") and state["runbooks"].hits:
        bullets.append(f"runbook match: {state['runbooks'].hits[0].title}")
    # confidence 强制设为 0.3——告诉下游"这是兜底，别 100% 信"
    return {"hypotheses": HypothesisList(hypotheses=[
        Hypothesis(title="Rule-based fallback", confidence=0.3, ...)
    ])}

这层在我们今天的本地测试里实际触发过：qwen2.5-coder:7b 输出的 JSON 被截断 → schema 解析失败 → fallback 用规则把 "deploy + runbook 命中 + 错误率飙升" 拼成一个 confidence=0.3 的假设。系统照样产出可用结果，只是带了"低置信"标记。

11自我改进闭环：L6#

系统不仅能跑，还会自己变好——而且不需要 prod 的真实流量来证明这一点。

问题的形态

L5 把"反馈飞轮"的管道全部建好了：每个 incident 记录用了哪个 prompt 版本（prompt_sha）、oncall 给了什么评价（thumbs_up/down + 文字订正）、harness 的每一次 LLM 调用都打了标签。管道是干的——没有 prod 流量，飞轮转不起来。

  2. 两个飞轮消费者——winner 提名 + 自动 runbook 起草——把数据变成可审查的 PR。

② Winner 提名（src/sre_agent/winner.py）

读 feedback 里的每条记录，按 (agent, prompt_sha) 二维分组，每组算 thumbs-up 率 + Wilson 置信区间，跑 two-proportion z-test。

三道提名闸门PYTHON
# 1. 样本量门槛：每个 arm 至少 50 条

    return hold("insufficient sample size")

# 3. 统计显著性：alpha=0.05 双尾
if p_value >= alpha:

return promote(winner.prompt_sha)

③ 自动 Runbook 起草（src/sre_agent/autorunbook.py）

真正学到东西的不是 thumbs-up——而是 oncall 给 thumbs-down + 写了订正的根因那一刻。那是"agent 错了，但人类正确"的训练对。

起草流程：

  1. 遍历每个 verdict ∈ {thumbs_down, incorrect} 且 correct_root_cause 非空的 feedback 记录。
  2. 按 (service, alert_shape) 聚类——alert_shape 是把 alert 描述抽干数字、去掉停用词、留头 4 个有意义词的 hash key。
  3. 每个聚类如果跨过 min_occurrences 阈值，emit 一段 Markdown：agent 说 X / oncall 说 Y / 建议动作。
  4. 挑出每个聚类里最常见的 remediation 作为"建议条目"。

关键设计：从不自动合并。起草器输出审稿用的 Markdown，真的 runbook 文案得人写——AI 写的 runbook 没有判断力，会把噪声当模式。

④ GitHub Actions：让飞轮自己转

L6 不靠人手动跑这些命令——靠定时任务。三个 cron，三条腿，看的是不同维度的反馈信号，开的是不同类型的 PR，过的是不同的 CODEOWNERS review。

Cron看什么数据改什么动哪个文件频率

harness-autorunbook thumbs-down 时 oncall 写的"正确根因" 加新知识 新建 runbooks/auto/*.md 每周一 02:00 UTC

每个 workflow 都接受 workflow_dispatch 手动触发——参数可以自定义（RNG 种子、A/B 流量比、显著性阈值），方便定向调查"上次 prompt 改动是不是把效果搞差了"。再叠一条 ci.yml 在每次 push/PR 上跑 pytest + ruff，保证这三个飞轮 job 本身的代码质量。

为什么是三个，不是一个？

"agent 这次没干好"这个事的根因，至少分三种——每种对应一种修复方式：

故障形态根因哪条腿管
同类问题里，新 prompt 答得更准 prompt 写得不够好 winner 提名换 baseline
oncall 反馈说"真正根因是 X"，但 runbook 库里压根没有 X agent 没见过这种故障 autorunbook 把 oncall 现场写下的修复方法整理成新 runbook
诊断本身没错，但 confidence 高了 / 低了，oncall 不知道该不该信 agent 自我评估不准 calibration 用真实 outcome 重拟合校准函数

分开做有两个工程上的好处：

  • 可解释性：PR 标题就告诉你"这次提的是 prompt 改动"还是"这次提的是新 runbook"——review 心智完全不一样。
  • CODEOWNERS 路由不同：prompt 改 sre-leads review；runbook 改服务 owner review；校准器改 sre-leads + ml-platform 双 review。

三条腿在做的事，更具体一点

① winner ——"哪个 prompt 真的更好用"

跑的是统计学：把同一个 agent 的不同 prompt SHA 分组，每组算 thumbs-up 率，跑两比例 z-test，Wilson 置信区间。

winner cron 输出REPORT

⏸ log-detective — hold
> not significant at alpha=0.05 (p=0.056) — collect more data

显著的开 PR 改 baseline 指针；不显著的 hold 着继续收数据。这条最重要的是它会说 "no"——一个总是 promote 的飞轮就是 p-hacking 机器，没用。

② autorunbook ——"我们少了哪几本 runbook"

聚类 thumbs-down 的反馈：按 (service, alert-shape) 把 oncall 的 correct_root_cause 字段聚成簇。簇里 ≥5 条 → 整理成一条新 runbook 草稿，PR 出去。

  Root cause: stripe webhook retry storm
  Mitigation: ...

review 通过合并后，下次同样的 alert 进来，runbook RAG 就能命中——agent 第一次就答得上。

③ calibration ——"85% 应该是真的 85%"

收集所有 (agent_confidence, oncall_verdict) 对，跑 PAV 等渗回归拟合一个分段函数。如果新 ECE 比线上低 ≥3pp 才开 PR：

| [0.80, 0.90) |     0.833 |        0.828 | +0.5pp |  ← 校准后

合并后，dashboard 上展示的 confidence 数字会自动经过这个函数——oncall 看到 "85%" 就真的是 85%，不用脑子里折扣 10pp。

用一个实际场景串起来

想象 oncall 这周给了 50 条反馈：

  • 25 条 thumbs-up，分布在 3 个 prompt variant 上 → winner 周日早上跑完，发现 variant 9a4e2b73 显著好，开 PR 换 baseline。
  • 12 条 thumbs-down，里面 7 条都说"真实根因是 stripe webhook 风暴" → autorunbook 周一早上跑完，开 PR 起草新 runbook runbooks/auto/stripe-webhook-storm.md。
  • 所有 50 条的 confidence 数字喂给 calibration 周日凌晨跑完，发现高分位 (>0.8) 普遍过自信 12pp → 开 PR 更新 data/calibrator.json。

三个 PR，三个独立 review，三个独立合并节奏。这就是"飞轮"两个字真正的意思——系统不是"上线就定型"，而是 oncall 每给一次反馈，三条腿就各自往前蹭一点。

② 三层 LLM fallback——主模型挂了不黑屏

开 SRE_LLM_FALLBACK=on，每次 get_chat_model() 返回的是一条链子，不是单个模型：

tier 链路DIAGRAM
primary (e.g. gpt-oss:20b, 30s timeout)
   ↓  on timeout / error / rate_limit
cheap (local ollama qwen2.5-coder:7b, 20s timeout)
   ↓  on timeout / error
rule-based (0-latency 兜底, schema-valid 占位响应)

每一次降级都会：

  • 写一条 LLMCallRecord(kind="fallback", agent=..., detail={from_tier, to_tier, reason})——可以在 /api/harness/calls 里事后回放。
  • 触发 sre_llm_fallbacks_total 计数器——监控直接可以看 "primary 在过去 5 分钟内挂了几次"。
为什么这事很重要

线上 LLM 不是"99.99%"那种 SLA。OpenAI 一次 5 分钟的 503 就能让你的诊断系统这 5 分钟里所有 alert 都黑屏。三层链子的代价是写 200 行代码，收益是把"主模型挂了"从 "incident" 降级成 "一个 fallback 计数器涨了"。

③ BM25 RAG + 持久化索引

  • 词频饱和：一篇 runbook 里出现 5 次 "OOM" 不等于出现 1 次的 5 倍权重。

G2 — Self-consistency 集成投票

论文：Wang et al. 2022 (Self-Consistency Improves Chain-of-Thought)。原理：跑 K 次 LLM 让它们彼此投票，挑最一致 / 最有信心的那个答案，准确率显著高于单次。

代码核心是个线程池——不是改 async。为什么？见 ADR-007：

  • 瓶颈是 LLM 服务端的队列深度，不是 Python 事件循环
  • 把 6 个 node 全改 async + LangGraph 改 astream + Flask 接 async 桥，工程量巨大、收益有限
  • 线程池 + ContextVars 快照：每个 worker 独立持有父线程的上下文（harness recorder 仍能正确归属）
# 单次（默认，K=1，跳过线程池开销）
SRE_HYPOTHESIS_ENSEMBLE_K=1 python ...

# 3 路投票（生产推荐）
SRE_HYPOTHESIS_ENSEMBLE_K=3 python ...
# Wall-clock latency ≈ max(member)，不是 3×；agreement 0.9+ 时基本无 hallucination

新指标 sre_ensemble_agreement——直接看模型有多稳：

0.9+    所有成员答的是同一个根因桶 → 输出可信
0.5-0.8 模型分裂 → picker 还是挑最 confident，但本身就是"硬 case"信号
< 0.4   每个成员都说不一样的 → 把置信度当作低，无论显示多少

16面试高频问答#

10 道一定会被问的，把答案背下来。

Q1. 为什么用 LangGraph 而不是直接用 LangChain agent？

LangChain agent（AgentExecutor）是循环+工具调用的抽象——LLM 自己决定下一步。这对发散任务好。
LangGraph 是显式有向图——拓扑代码里画明白。事故响应是 SOP（固定流程），用 LangGraph 拓扑可预测、可测、能加 checkpointer 做断点续跑。
另外，LangGraph 是 LangChain 团队自己造的，明确定位为"agent 编排"——LangChain agent 已经是 legacy 方向。

Q2. 8 个 agent 共享什么状态？怎么避免竞争？

共享 GraphState（一个 TypedDict）。LangGraph 的关键设计：每个节点的输出是个 dict patch，框架自动合并到 state。同一批并行节点的 dict key 不能冲突——这是 lane discipline 的另一个落地：每个 worker 只写自己的字段（log_detective 只写 state["logs"]）。
LangGraph 用 reducer 合并不同字段，不会有并发覆盖。

Q3. 你的系统能扛 TikTok 量级（每分钟一千万条日志）吗？

关键认知：agent 不直接处理一千万条原始日志。Telemetry 后端（ByteLog/ClickHouse）已经把原始数据压成"过去 15min 错误 top-10"这种百行级摘要，agent 只看摘要。
规模化挑战不在"agent 处理速度"，在三个地方：(1) 告警突发（用 Kafka + Temporal） (2) LLM 成本（tier 路由分流） (3) 后端查询延迟（recording rules + 高基数控制）。这些都在 README Phase E roadmap 里。

Q4. LLM 输出格式错了怎么办？

5 层防御（第 9 节讲过）。最关键的是每个 LLM agent 都包 try/except + rule fallback——LLM 挂了就走启发式拼一个 confidence=0.3 的诊断，标低置信度透传给人。
今天本地测试就实际触发过：qwen2.5-coder:7b 的 JSON 被截断 → schema 解析失败 → fallback 接住 → 仍然产出 5 路证据 + 3 条修复建议。

Q5. 你怎么测试一个 LLM 系统？非确定性怎么办？

分三层：

  • 单测：mock 掉 LLM provider，跑每个 agent 的"纯逻辑"部分（schema 校验、降级路径、provider 调用次数）。130 个测试全 offline 跑，不依赖任何外网
  • 合约测试：每个 agent 的输出必须满足 Pydantic schema，schema 即合约
  • 评估集（roadmap）：10-20 个历史真实 incident + 人工标注的正确根因，定期跑，正确率 ≥ 80%

非确定性：用 temperature=0 + 强 schema + 关键字 assertion，不假设字面文本相同。

Q6. 你为什么有 1 个 agent 不用 LLM？

runbook_consultant 的工作是"从 wiki 里检索相关段落"——这是纯检索任务。用 LLM 来做反而：(1) 慢（多一次 API 调用）(2) 烧钱 (3) 可能幻觉出根本不存在的 runbook 路径。
用 RAG（向量相似度搜索）就够了。它的输出再喂给 hypothesis_generator 这种真正需要推理的 agent。这就是 "用对工具" 而不是 "啥都用 LLM"。

Q7. tier 路由是怎么决定的？谁来分？

用 classify_tier()，输入 severity / description / 是否命中 runbook / 是否有强信号，输出 rule / cheap / premium。
规则（精简版）：

  • 合成测试流量、明显误报 → rule（不调 LLM）
  • SEV-1 / P1 → premium（值得贵模型）
  • SEV-2 但没 runbook 命中 → premium（信息不足时让大模型推理）
  • 其他默认 → cheap（占大头）

今天分类是 visible cosmetic（UI 显示），明天接真模型端点就是真路由。

Q8. 怎么换 LLM provider？比如从 OpenAI 切到 Claude？

改一个环境变量：SRE_LLM_PROVIDER=anthropic。
工厂模式 get_chat_model(role) 内部根据 provider 选 LangChain 的 ChatOpenAI / ChatAnthropic / ChatOllama 实现。Agent 代码一行都不动——这就是工厂模式的价值。

Q9. 假设 oncall 觉得诊断错了，怎么反馈？

Roadmap 里有"Slack action buttons"——每条 incident 推送时附 👍 / 👎 / 修改 按钮，点击 POST 回 /api/incidents/<id>/action。
更深一层：累积的负反馈会自动生成"建议 runbook 草稿"——下次类似 alert 命中就走对。这就是团队大脑越用越聪明的飞轮。

Q10. 这套系统跟传统 SRE 工具（如 PagerDuty AIOps）比有啥独特？

传统 AIOps 大多是统计模型：相关性分析、异常检测、聚类去重。它擅长"哪几个告警是一回事"，不擅长"根因是啥"。
LLM 系统的核心增量是把团队 tribal knowledge（runbook、wiki、Slack 历史）变成可查询的语义索引，让诊断不再依赖凌晨 3 点叫醒资深工程师。
这两者其实互补不冲突——统计模型负责"分组+优先级"，LLM 系统负责"解释+建议"。一个完整的 SRE AIOps 栈两者都要有。
